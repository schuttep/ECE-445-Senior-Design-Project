# 2026-02-15 - Figuring Out the Display Connector
I have a DSI touch display from a previous project that originally connected to a Raspberry Pi via ribbon cable. I spent time figuring out whether we could use this display directly with the ESP32. The ribbon cable on this display uses a GDI port, which the ESP32 does not support natively and would require a separate bridge chip to use. After looking more carefully at the display's PCB, I found that it also has broken-out pads for SPI, I2C, and several analog control pins. This meant I could wire it directly to the ESP32 with jumper wires over SPI, avoiding the bridge chip entirely and keeping the design simpler. I confirmed the display used an ST7365P controller, which had an existing Arduino library (DFRobot_GDL) I could build on top of rather than writing a low-level driver from scratch.

# 2026-02-23 - Initial Breadboard Setup
Met with the team to assemble the breadboard ahead of the upcoming demo. We wired up 8 Hall effect sensors at 1.5" center-to-center spacing, which matched the tile dimensions we were planning for. We soldered the ADS7128 to its QFN breakout board and connected it to the ESP32 dev board over I2C. We hit issues initially where the ESP32 could not communicate with the ADC at all. The root cause was a combination of the wrong I2C address in our code (the ADS7128 default address depends on the ADDR pin configuration) and a cold solder joint on one of the breakout pads that we found and reflowed. Once both were fixed the ADC responded correctly on the bus. The magnets we had were weaker than expected and required nearly direct contact with the sensor to produce a reading above the noise floor. This was a concern but we noted it and ordered stronger magnets as a follow-up. After getting the hardware stable I started on the LED code so we could show individual sensor responses visually during the demo.

# 2026-03-25 - WiFi and API Driver Bringup
With the main PCB validated on the hardware side, I started writing the WiFi and API connection code targeting the ESP32-S3. The goal was a clean layered design where the game loop would not need to know the details of the network transport. I split this into two layers: a low-level `wifi_driver` that handled connecting to a network by SSID/password using the ESP32 WiFi library with a retry loop up to 40 attempts, and a higher-level `api_connect` layer that handled HTTPS communication with our AWS API Gateway backend. Because the backend ran over HTTPS, I needed to configure `WiFiClientSecure` with the Amazon Root CA 1 certificate so the ESP32 could validate the server's TLS certificate rather than just skipping verification. I hardcoded the PEM-encoded Amazon Root CA directly in the source since the ESP32 does not have a system certificate store. The API layer exposed functions like `fetchGameState()`, `pushFENState()`, `sendHeartbeat()`, and `resetGame()` so the game loop could stay focused on state transitions instead of HTTP details.

# 2026-03-27 - Programming the Main PCB For the First Time

This confirmed the PCB layout, routing, and component placement had translated correctly from the breadboard prototype. # 2026-04-07 - Display Interaction and WiFi Manager UI
This was the first major UI-heavy milestone. I spent a lot of time making the display easier to interact with and building out the display-side abstractions I would need for menus and game status screens. The display uses the ST7365P controller over hardware SPI and I initialized it through the DFRobot_GDL library. The first issue I ran into was orientation: the display was initializing in portrait mode by default and rendering everything rotated 90 degrees from what the player would see, so I fixed that and moved the whole interface into landscape mode.

I then built the board renderer on top of the driver. The board is drawn as an 8x8 grid of 32x32 pixel squares with cream and brown alternating colors. Each piece is rendered as a filled circle with the piece letter centered inside it, white pieces getting a white disc with a black rim and black pieces getting a black disc with a grey rim. To avoid redrawing the entire board every loop iteration, I added a per-square cache that tracks what was last drawn in each square and a 64-bit dirty-cell bitfield for squares that need to be forcibly repainted after highlights or overlays. I also added an early WiFi manager UI so the player could scan nearby networks from the screen instead of hardcoding everything into the firmware every time we changed environments.

# 2026-04-16 - ADC Testing in Main Flash
Added simpler ADC test support directly into the main firmware image so we could test real PCB hardware without constantly swapping sketches. The board uses eight ADS7128 chips addressed 0x10 through 0x17 over a dedicated I2C bus (`Wire1`) at 100 kHz. Each chip handles one column of 8 squares, and because the physical routing order did not match logical file order, I added a lookup table to map file `a` through `h` to the actual chip wiring order `{7, 6, 5, 4, 0, 1, 2, 3}`.

Each channel is read by putting the ADS7128 into manual mode, selecting the channel, waiting briefly for the conversion to settle, and reconstructing the 12-bit reading from the two returned bytes. I set the baseline to 2048 and used a threshold of about 300 counts above or below baseline to determine whether the magnet polarity indicated a white piece, black piece, or empty square. Adding these checks directly into the main flash made hardware debugging much faster during assembly and validation.

# 2026-04-21 - Full System Integration
This was the point where the major pieces finally connected. I integrated the ADC board reading path, the display renderer, the WiFi/API layer, and the move validation logic into a single control flow. The big addition here was the finite state machine in `gameloop.cpp`, since gameplay has too many distinct waiting conditions to manage safely with just a few flags in `loop()`. I defined states for waiting for game start, joining a game, initializing state from the server, syncing the physical board, waiting for a local move, validating it, confirming it on the touchscreen, sending it to the server, waiting for the remote move, applying the remote move, and handling game-over conditions.

The game manager tracks committed FEN, pending FEN, castling rights, en passant state, half-move clock, server versioning, timers, and AI difficulty. This was the first point where I could reasonably say the final architecture existed, even if it still needed a lot of refinement.

# 2026-04-22 - Physical-To-Logical FEN Translation
One of the harder parts of the project was translating a raw physical board reading into a valid logical chess FEN. The ADC driver only knows occupancy and polarity, not piece identity, so a Hall effect reading can tell me that a white piece or black piece is present on a square but not whether that piece is a rook, bishop, or queen. I solved this with `cgm_physicalToLogicalFEN()`, which takes the last committed logical FEN and a new physical reading and reconstructs the move.

The algorithm works by first converting the committed FEN into an expected physical board using piece polarity, then comparing that expected board against the new physical reading. If too many squares changed, it rejects the reading as ambiguous. If only departure squares changed and no arrival square appeared, it treats that as a piece-in-air case and waits. For normal moves it clears the departure squares and then maps arriving squares back to the most likely committed source pieces of the correct color. It also has explicit handling for castling and correctly supports en passant by recognizing the capture square disappearing separately from the arrival square. This was a major milestone because it let the board infer real chess moves from only magnet polarity and prior board state.
